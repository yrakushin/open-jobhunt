# Методология оценки соответствия вакансии (match)

Версия: 2026-07-03. Используется в Open JobHunt для решения «откликаться / пропустить».

## 1. Зачем

Плоский процент «75% match» без структуры даёт ложные пропуски: вакансия с 5 «желательно» и 1 «обязательно» оценивается так же, как 6 «обязательно».  
Цель — **взвешенное сопоставление** профиля и вакансии с разделением критериев по силе требования.

Опирается на практики из open-source ATS/matching (weighted skill scoring, must/preferred/bonus tiers):

- [placement-ops](https://github.com/shotwellj/placement-ops) — taxonomy, required/preferred/nice-to-have, blockers
- [resume_job_scoring_engine](https://github.com/farhan-439/resume_job_scoring_engine) — multi-dimensional weights
- [ai-resume-screener](https://github.com/mohamed-elkholy95/ai-resume-screener) — required 70% + preferred 30% в skill_score

## 2. Шаги анализа вакансии

### 2.1. Извлечь критерии из текста

Источники (даже если HR не разметил секции явно):

| Сигнал в тексте | Тип критерия | Примеры формулировок |
|-----------------|--------------|----------------------|
| «обязательно», «must», «требуется», «необходим», «без этого не рассматриваем» | **must** | «обязателен опыт PM 3+ года» |
| «важно», «нужен опыт», перечень в блоке «Требования» без «будет плюсом» | **must** | «опыт с LLM-продуктами» |
| «будет плюсом», «желательно», «prefer», «nice to have», «преимуществом» | **preferred** | «знание SQL — плюс» |
| «идеально», «strong plus», «отлично если» | **bonus** | «опыт в fintech» |
| Явный стоп для профиля | **blocker** | «только ML Engineer», «стажёр», «5+ лет PM» при 3.5 у кандидата |
| Уровень / роль | **must** (высокий вес) | product vs project vs sales |
| Домен / стек | **must** или **preferred** | AI/LLM, B2B enterprise |
| Опыт (лет) | **must** | «от 3 лет» — must; «от 5 лет» при 3.5 — blocker или must с низким match |
| Формат / локация / зарплата | **must** (если явно) | уже частично в pre_filter |

Если секции «Требования / Будет плюсом» нет — классифицировать **по лингвистическим маркерам** и контексту (глагол «требуется» vs «желателен»).

### 2.2. Назначить вес каждому критерию

Базовые веса (сумма нормализуется):

| Тип | Базовый вес | Смысл |
|-----|-------------|--------|
| blocker | — | Любой с match=0 → вакансия **отклоняется** |
| must | 3.0 | Обязательное; провал → cap score или отказ |
| preferred | 1.5 | Желательное; влияет на score, не блокирует |
| bonus | 0.5 | Плюс; не штрафует сильно за отсутствие |

Дополнительно для конкретной целевой роли (берётся из `context/resume-profile.md`):

- must ×2: product ownership (не project-only), уровень (не junior)
- must ×1.5: ключевые домены/технологии кандидата **если в заголовке или явно в must-блоке**
- preferred: смежные навыки и домены из профиля
- blocker: чистый IC (DS/ML engineer), sales/HR/marketing PM masquerade

> Конкретные роли/домены/стоп-роли задаются в профиле пользователя
> (`context/resume-profile.md`). Методология универсальна.

### 2.3. Оценить match по каждому критерию (0.0–1.0)

| match | Значение |
|-------|----------|
| 1.0 | Прямое подтверждение в профиле |
| 0.7 | Смежный опыт (AI workspace vs AI product; B2B vs enterprise) |
| 0.4 | Частичное / слабое |
| 0.0 | Нет / противоречит профилю |

### 2.4. Формула итогового score

```
weighted_sum = Σ (weight_i × match_i)
total_weight = Σ weight_i
raw_score = 100 × weighted_sum / total_weight
```

**Правила отсечения:**

1. Любой **blocker** с match < 0.5 → `score = 0`, причина «blocker».
2. Любой **must** с match < 0.4 → `score = min(score, 49)` (ниже порога отклика).
3. Два и более **must** с match < 0.5 → пропуск независимо от raw_score.

**Пороги отклика (рекомендация):**

| Score | Действие |
|-------|----------|
| ≥ 75 | Сильное совпадение — отклик |
| 60–74 | Отклик, если нет провалов must |
| 50–59 | Пропуск (можно в отчёт «на грани») |
| < 50 | Пропуск |

Дефолт в config: `min_match_percent: 65` (не 75).

### 2.5. UI: 3 кнопки (Open JobHunt)

В веб-интерфейсе — **одна строка из трёх кнопок** (пресеты):

| Кнопка | Порог | Критерии |
|--------|-------|----------|
| Низкий | ≥10% | все критерии на «низкий» (≥15%, bonus) |
| Средний | ≥55% | все критерии на «низкий» (≥15%, bonus) |
| Высокий | ≥65% | смешанный пресет (роль/опыт high, b2b low…) |

Сохраняется в `config.yaml` → `match:`.

## 3. Связь с pre_filter (жёсткие правила до LLM)

До weighted match уже отсекаются:

- стоп-роли (junior, стажёр, devrel…)
- чёрный список / пауза компаний
- зарплата < min_salary_rub (если указана)
- дубли откликов

LLM-match **не дублирует** эти проверки — только содержательное соответствие.

## 4. Тесты / опросы / friction

Отдельно от match-score, **до отклика** помечать:

- «пройти тест», «опросник», «assessment», «тестовое задание»
- внешняя ссылка на анкету

В логе: `⚠ тест/опрос — отклик возможен, тест вручную`.  
В отчёте: колонка/причина `требует тест`. Отклик **не блокируется**, но пользователь видит такие кейсы.

## 5. Ограничения скана вакансий (отдельная проблема)

hh.ru не отдаёт «все вакансии сайта» — только:

1. **Подбор по резюме** (`search/vacancy?resume=…`) — обычно 1–3 тыс. релевантных hh.
2. **Keyword-поиск** — дополнение по queries.

Лимит `max_vacancies_to_scan: 0` = **без обрезки** (раньше 2000 — из-за этого «~1800»).  
`max_pages_resume` — сколько страниц подборки по резюме листать (100 стр × 100 = до 10 000 id).

## 6. Калибровка

Раз в 1–2 недели сверять:

- отклики с match 60–70 — были ли отказы/интерес
- пропуски 70+ — были ли бы релевантны
- корректировать веса must для AI/PM в `prompts/match-scoring.md`
